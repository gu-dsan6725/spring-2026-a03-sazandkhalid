# Question 4: How does the authentication flow work, from token validation to user authorization?

Query Category: `Security/Auth`

The authentication flow is a multi-step process involving an internal Auth Server and Keycloak:

1.  **Token Issuance**: Users authenticate via Keycloak (referenced in `init-keycloak.sh`). Upon success, an OAuth2/OIDC token (JWT) is issued.
2.  **Request Handling**: Incoming requests pass through an Nginx reverse proxy (`mcp-gateway-registry/docker/nginx_rev_proxy_http_and_https.conf`).
3.  **Token Validation**: The proxy or the `auth_server/server.py` validates the JWT signature and expiration.
4.  **User Authorization**: Once the token is validated, the system extracts claims (scopes/roles). The registry service then uses these claims to authorize access to specific API endpoints.

**Citations**:
- `mcp-gateway-registry/auth_server/server.py`: Handles core authentication logic.
- `mcp-gateway-registry/docker/nginx_rev_proxy_http_and_https.conf`: Manages proxy-level auth.
- `mcp-gateway-registry/terraform/aws-ecs/scripts/init-keycloak.sh`: Configures the IDP.

# Question 5: What are all the API endpoints available in the registry service and what scopes do they require?

Query Category: `API/Endpoints`

The registry service exposes several critical endpoints, primarily managed through specialized routers:

**General Service Endpoints**:
- `GET /health`: Health check.
- `GET /api/auth/me`: Current user session info.

**Registry & Peer Management (from `registry/api/`):**
- `POST /api/peers/{peer_id}/sync`: Synchronizes registries between peers. (Requires write scopes).
- `GET /api/peers/{peer_id}/status`: Checks sync status. (Requires read scopes).
- `POST /api/peers/{peer_id}/enable`: Enables a peer registry.
- `GET /api/peers/{peer_id}/shared-resources`: Lists shared MCP tools/resources.

**Citations**:
- `mcp-gateway-registry/registry/main.py`: Top-level endpoint definitions.
- `mcp-gateway-registry/registry/api/peer_management_routes.py`: Sync and status endpoints.

# Question 6: How would you add support for a new OAuth provider (e.g., Okta) to the authentication system? What files would need to be modified and what interfaces must be implemented?

Query Category: `Security/Auth`

Adding a new OAuth provider like Okta involves extending the authentication server's provider layer:

1.  **New Provider Implementation**: Create a new class (e.g., `OktaProvider`) in a file similar to `mcp-gateway-registry/auth_server/providers/cognito.py`.
2.  **Interface Requirement**: Must implement the standard OAuth2 interface, including methods for `get_auth_url`, `exchange_code_for_token`, and `validate_token`.
3.  **Configuration**: Modify `mcp-gateway-registry/auth_server/server.py` to register the new provider and add Okta credentials to the env configuration (referencing `.env.example`).
4.  **Frontend Update**: If applicable, update `mcp-gateway-registry/registry/templates/login.html` to include the Okta login option.

**Citations**:
- `mcp-gateway-registry/auth_server/server.py`: Main auth server coordination.
- `mcp-gateway-registry/auth_server/providers/`: Directory containing provider implementations.
